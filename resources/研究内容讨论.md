## 研究内容讨论

**任务定位(固定)**

- 对象:多源异构噪声序列数据(JSON 树型 / CSV 表格型 / SQL 关系型),文件内结构同质、实例异质、局部结构受损
- 任务本质:**结构退化条件下的字段语义类型标注(CTA 式,主)+ 值级 span 检测(NER 式,辅)**,非纯 NER。
- 三层噪声(按破坏的结构层次,以标准解析器行为为判据):**形式层**(解析器无法解析:截断、引号/括号不闭合、分隔符未转义);**结构层**(可解析但 schema 偏移:列漂移、嵌套错位、记录边界混排);**语义层**(可解析、结构正确但字段含义不定:无表头、通用化命名、缩写、同名异义/异名同义)

**研究内容一**（结构表示）已确认架构：

- 基座:代码/结构预训练编码器(GraphCodeBERT / UniXcoder),在 IR 数据集上适配(全量或 LoRA)

- 输出两个:$H_{fld}$(字段稠密表示) + $c_{struct}$(字段级结构置信度,有监督头,校准较好)。样本级 $\hat{I}(x)$ 头已砍(解析器能算,神经网络预测它冗余)

- 辅助头 $p_{type}$:塑造 $H_{fld}$,内容二里复用为"编码器路径直判"

- 以结构/代码预训练编码器为主干，采用 LoRA 进行参数高效适配，其上接入字段级敏感类型分类头、结构置信度头。标准做法恰恰是两者同时用：**主干用 LoRA 做参数高效适配（只训低秩增量）**，任务头（分类头、置信度头、回归头都属于此）作为新增参数**全量训练**。

- 结构鲁邦感知编码器：输入侧：

  通道一：容错解析可恢复的字段单元：经过IR容错解析层后得到的SchemaUnit 线性序列化为序列单元；

  通道二：不可恢复的区域以原始文本片段保留进输入，训练时额外附带对齐的干净版本$x$、各噪声标签。

- 训练目标 $\mathcal{L} = \mathcal{L}_{type}(x') + \lambda_1\mathcal{L}_{struct\text{-}conf}(x') + \lambda_2\mathcal{L}_{consist}(x, x')$;其中 $\mathcal{L}_{consist}$(干净 $x$ 与带噪 $x'$ 一致) = NAT stability training,**噪声鲁棒性真正来源**;监督全部来自注噪,零人工标注

- 双通道输入:通道一 = 容错可恢复字段单元(IR 序列化),通道二 = [RAW] 不可恢复原文。两通道 = $I(x)$ 划分的两侧;干净时通道二为空(优雅退化);注噪本身生成双通道结构

- 编码器训练完后冻结交给内容二

**研究内容二**（鲁棒识别）已确认架构：

- 五级主脊:候选检测(高召回) $\rightarrow$ 结构上下文校正 $\rightarrow$ 结构置信度感知的多证据融合 $\rightarrow$ 置信度驱动分流 $\rightarrow$ 实体级关联融合

- 融合:$s = \psi(s_{value}, c_{struct} \cdot s_{schema}, c_{struct} \cdot s_{context}, s_{cross})$

- LLM 协同 = 结构即模态(把 $H_{fld}$ 经投影器注入 LLM 嵌入空间作软 token,GraphToken/LLaVA 式),保留可读文本的混合通道,非纯提示词工程;大模型作为融合里一路证据,非难字段最终裁决者

- 投影器对齐两阶段:Stage 1 冻编码器+冻 LLM、只训投影器(任务"结构向量 $\rightarrow$ 描述",描述从干净 SchemaUnit 自动生成,免费);Stage 2 训投影器 + LLM-LoRA 做难字段识别

- 噪声驱动路由:按结构完整性路由,主轴 $c_{struct}$,次级 = 识别置信度 + 证据冲突;边界靠"可靠性曲线 $\times$ 升级增益曲线 $\times$ 成本前沿"在验证集标定(升级仅在"编码器不可靠 $\land$ 大模型确实更好"处)

- 资源:开源 LLM + 1-2 $\times$ A100 80G

- 关键先例:GraphToken(2402.05862)、LLaVA(2304.08485)、BLIP-2、NAT(ACL2020)、CTA 谱系(Sherlock/Sato/DoDuo/RECA)、RobuT、FrugalGPT/RouteLLM

**研究内容三**——待定：倾向:折叠进内容二作"成本/置信度感知分流"(选项2),或重定义为系统级效率层(选项3)

**当前数据状态与问题**：

- 真实格式分布:free_text 67.3% / json 22.1% / csv 8.6% / sql 1.2% / 其余<1%(此分布本身是"真实数据湖结构化仅约 32%"的一手证据)

- 当前嗅探:Tier1≈3000,Tier2≈700(T2 偏低可疑,大概率是 resync 缺失把可恢复文件冲进 T3、饿死了 T2;修复后预测 T2 显著上升,这本身是验证$I$实现的 sanity check)

- 已识别$I$实现 bug:CSV I≡1.0(good/total 无条件同步自增)、JSON 早期崩溃后后续 good record 不计、SQL 把 DDL/DML 纯度混进$I$、64KB 头采样对尾部截断盲视

数据预处理流程：

- **数据划分**:Tier1(干净)→注噪种子→训练集;Tier2(真噪)→迁移测试 + 噪声频率标定;Tier3→排除。Tier1 要纯度(precision),Tier2 要真实噪声刻画准
- **退化曲线**:合成路线(受控注噪),按合成样本上实测 II I 分桶(或按注入强度)

- 可控注噪在每一次的噪声算子施加后，自动获得一个对齐三元组：（干净样本、带噪样本、噪声元数据——噪声做了什么改动、如何改动等信息、属于哪一层的噪声破坏、字段路径是否被改动）这些信息都是**编码器所有输出头的训练标签来源**。属于是有监督的。



**研究内容一与研究内容二的结合处：**

结构即模态——把结构感知编码器的输出作为一种模态,经学习投影注入大模型,用于噪声鲁棒的敏感信息抽取，参照**GraphToken**；沿用**LLaVA**两阶段。

重点在于编码器的输出与大模型输出的投影对齐问题和任务微调，需要接入MLP投影层，将编码器的输出嵌入到LLM嵌入空间中。

**Stage 1 — 对齐(冻结编码器 + 冻结 LLM,只训投影器)。** 任务是"结构向量→文本描述",即给 `<struct>`(某字段的 HfldH_{fld} Hfld),让 LLM 输出该字段的结构描述,例如"路径 order.buyer.tel,buyer.contact 下的叶子,兄弟字段 name、addr,值形如电话号"。损失就是标准 LM next-token loss,梯度回传到投影器。

**Stage 2 — 任务微调(训投影器 + 给 LLM 套 LoRA,编码器可冻结)。** 任务是真实的难字段敏感识别,输入 = 结构软 token + 可读文本(值/路径/邻域),用你实际的(较少的)敏感标签。Stage 1 已把投影器预训好,所以 Stage 2 需要的标注数据更少。



 每个字段是否升级到大模型,由内容一来的结构信号决定——cstructc_{struct} cstruct(结构置信度)、识别置信度、证据冲突程度。判据是:结构可信 + 编码器判得确定 + 证据一致 → 留在廉价编码器路径;结构置信度低(schema 路径被破坏)、或识别置信度低、或证据冲突(值像手机号但路径说它是 ID)→ 升级到大模型

**训练过程**

**相位 A(内容一)编码器适配**:以代码/结构预训练编码器(GraphCodeBERT/UniXcoder)为基座,在内容一自己构造的 IR 数据集(干净 unit + 其注噪版本)上适配训练,得到一个冻结的、噪声鲁棒的结构编码器,产出输出。这就是内容一的全部训练,此后编码器冻结。

**相位 B(内容二 Stage 1)投影器对齐**:冻结编码器(来自 A)+ 冻结 LLM,只训投影器。

**相位 C(内容二 Stage 2)任务微调**:冻结编码器,训投影器 + LLM 的 LoRA(+ 一个轻量融合模块)。

### **双通道输入更正**(待讨论确认)

双通道不是指双塔、两个编码器，基座使用 GraphCodeBERT / UniXcoder,它们的原生输入是一条 token 序列,结构是靠 attention mask（把结构当边喂） 和 special token /segment embedding 注进去的,不存在两路独立编码器:   - GraphCodeBERT：[CLS] 代码token序列 [SEP] 变量/数据流节点,数据流边经 graph-guided masked attention 注入; UniXcoder：把 AST 拍平进同一条序列,用前缀控制 token 区分模式。

文件每一段按"容错解析器能否恢复结构"路由到两种表示:   

-  通道一：内容：可恢复字段单元{path，sample，dtype}；任务：CTA 式字段语义类型标注；有结构边（需要路径/拓扑）
- 通道二：内容：不可恢复的原始文本span【RAW】；任务：值级span检测，无结构边

注噪本身生成双通道结构，但只有形式层噪声会把内容从通道一推进通道二；

 - 形式层(截断 / 引号·括号不闭合 / 分隔符未转义)→ 让某区域解析器解不动 → 该区域从通道一掉进通道二    [RAW]。这是"双通道"的唯一生成机制,也是"什么该进通道二"的监督来源。                                  
  - 结构层(列漂移 / 嵌套错位 / 边界混排)→ 仍可解析,留在通道一,但路径错了 → 喂 c_struct / 类型头的损坏标签。                                                                                  
  - 语义层(改名 / 通用化 / 去表头)→ 留在通道一,名/义被污染,结构完好。

所以"双通道划分"≈"形式层是否命中"的二值后果;另外两层根本不动通道归属。**这点讨论稿和开题中为明确说明。**

**双通道序列怎么拼**：seg1包含结构边信息，经 graph-guided attention mask注入；seg2要保位置，让坏的span紧挨结构邻居，即【RAW】挂最近可恢复祖先路径做锚点。

**双通道序列怎么对齐？**：把干净字段用形式层噪声打成[RAW] 时,有监督信号注噪信息 oracle,我们知道它原来的 path / dtype / 值 / 拓扑。于是在 [RAW] token上挂结构恢复辅助头,标签全来自噪声元数据:                                                                                                                                                    

  - 潜结构路径头(latent-path):对 [RAW] 内 token 预测其所属字段路径(标签=被破坏字段的原 path)→ 教编码器在原文上重建路径;                                                                           

  - 边界/切分头:预测 [RAW] 内字段/键值边界(解析器切不出,oracle 知道);                                 

  - 再锚定/

      结构链接头:预测恢复片段挂在哪个 seg1 节点下。

  - 锚点默认在形式层噪声截断前最后一个成功解析的结构位置，

于是 [RAW] 上同时跑:**结构恢复(路径/边界/链接)+ 值级 span 检测**。NER 只是其中一个头。

  (2) 训练期前向(每个样本)

1. Tier1 干净文件 → 解析成全结构(全 seg1)= x;
2. 施加噪声算子 T(x, Θ_n)(形式层会制造不可解析区)→ x';                                              
3. 对 x' 重跑容错解析器 → 恢复出的字段(seg1)+ 解不动的 【RAW]】seg2)。这一步重解析才是 x' 上 I(x)划分的真正产生处;                                                                             
4. 线性化 x'(上面的格式);另起一遍线性化干净 x(全 seg1);
5. 两条序列各过编码器 → 逐字段 pool 出 H_fld,c_struct / p_type 头从字段表示读出;
6. 损失:L_type(x')(类型,标签来自 pii_seed 弱标/oracle)+L_struct-conf(x')(c_struct,标签来自噪声元数据:该字段属哪层破坏)+ L_consist(x,x')(干净/带噪字段表示一致,NAT 稳定性,鲁棒性真正来源);                                               
7. 回传到 LoRA + 各头。

推理期前向(真实数据)                                                                                                                                                                           解析器跑真实文件 → I(x) 划分(此处是估计)→ 同样线性化 → 前向得 H_fld + c_struct。c_struct 低的字段(结构疑似被破坏)交内容二升级 LLM;[RAW] span 上编码器做值级 span 检测,捞回坏区里的 PII。

问题：

 **L_consist 要按字段身份(path)对齐,而非按位置:形式层噪声会把同一个字段从 x 的 seg1(结构单元)变成 x'的 seg2([RAW] 里的一段)。一致性损失必须能跨"通道重分配"把这两种表示对上 → 要求 [RAW] span 携带    provenance(它覆盖了哪些干净字段),用 path 作锚。这点不解决,L_consist 在形式层噪声上根本算不对。**

claude code 回答：按照这个逻辑，在喂给编码器的输入质量还得依赖于容错解析器的实现性能？ 

**是,而且这是这套架构的定义性质,不是实现瑕疵。** 编码器输入 = 通道一(容错可恢复的 SchemaUnit IR)+ 通道二(`[RAW]` 不可恢复原文),而**两通道的分界线就是容错解析器的恢复边界**(讨论稿:双通道 = $I(x)$ 划分的两侧)。容错恢复得多,通道一就大;恢复得差,通道二就大。它直接塑形编码器看到的东西。

但"依赖"要理解准,否则会得出过度悲观的结论:

**1. 它移动的是两通道间的"负担",不是系统能力的天花板。** 容错强 → 更多落通道一(干净 IR,编码器轻松);弱 → 更多落通道二(原文,编码器多干活)。但两种情况系统都能跑,**因为编码器本来就是在通道二上训练的**——读不可恢复原文正是它的职责(干净时通道二为空 = 优雅退化)。所以容错解析器是个**成本/难度旋钮**,不是"解析器多强、系统上限就多高"。把它当 tokenizer 看:切得好不好影响下游难度,但模型不被 tokenizer 卡死。

**2. 真正要盯的不是"质量高低",是两件更要命的事:**

**(a) 一致性——训练、推理、各阶段必须用逐字节相同的容错函数。** 编码器学的是"给定这个容错解析器,通道一/二各长什么样"的统计;换个恢复行为不同的解析器,通道分布就漂 → 训练/推理错配。这跟"tokenizer 训推必须同一个"是一回事。**这个坑你代码里已经踩到并在打补丁**:`schema_partition._stream_json_records` 的注释明说要让分片阶段的 json/json5 容错"与 parse 阶段一致",否则"只靠容错进 tier1 的 JSON 在分片阶段零产出"。这正是同一段容错恢复被两处各写一遍、要人肉保持同步的征兆。→ **应把容错恢复(含通道切分)抽成唯一一个共享函数**,parse 算 $I$、extract 分片、将来组装编码器输入全调它,否则阶段间必漂。

**(b) 失败模式不对称——宁可少救,不可错救。**

- 欠恢复(可救的丢进通道二)= 安全:编码器当原文读,只多花力气。
- 过/错恢复(把列对齐错位、吃掉 `;` 的合并语句,当成"恢复成功"塞进通道一)= 危险的**静默污染**:编码器把通道一当"较干净"来信任,却被喂了带结构标签的垃圾,分辨不出。

这对应上一轮的 $P$(repaired):**$P$ 只能计你信得过的恢复**,拿不准的算法上必须归通道二($L$),不是通道一。容错解析器要设计成"不确定就降级到原文",而非硬凑一个自信的错结构。

**3. 架构自带对冲:$c_{struct}$ + 对齐三元组,让编码器对"自己解析器的错"也鲁棒。** 编码器不盲信通道一。因为你有对齐三元组(干净 $x$、带噪 $x'$、元数据),**训练时你确切知道容错解析器在 $x'$ 上的恢复哪里偏离了真值 $x$** → 就在那些字段把 $c_{struct}$ 监督成低。于是编码器学到的不只是"对噪声鲁棒",而是"对我这个容错解析器的恢复错误也打折扣"——这正是 $c_{struct}$ 的本职,也是这套方法比"光一个容错解析器"多出来的部分。配套:让容错解析器**本身输出每单元的恢复置信度**,作为通道一质量信号去喂/初始化 $c_{struct}$。而且因为有 $x$ 真值,你能在合成集上**直接度量容错恢复的 precision/recall**,把它当一根消融轴。

**4. 于是"质量"问题被转成"成本"问题,正好接内容三。** 容错越弱 → 通道二越大 + 低 $c_{struct}$ 字段越多 → 升级大模型的比例越高 → 在内容三体现为**单位样本成本上升**。所以容错解析器的性能主要是一根**效率杠杆**(决定工作落在廉价编码器路径还是昂贵 LLM 路径),其唯一的正确性风险(静默错恢复)由第 3 点中和。换言之:投入在容错解析器上 = 降推理成本,而不是"补正确性"。

------

一句话:是的,编码器输入质量挂在容错解析器上——但它是**可冻结、可度量、可被 $c_{struct}$ 兜底**的预处理层,移动的是成本与负担分布;唯一不能松的两条是"训练/推理/各阶段同一个容错函数"和"宁可降级到原文,也不要自信地错恢复"。
